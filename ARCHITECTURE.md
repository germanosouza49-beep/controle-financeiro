# FinControl -- Arquitetura do Sistema

## 1. Visao Geral

FinControl e um monorepo com 4 modulos independentes que se comunicam via HTTP (REST API) e compartilham tipos TypeScript.

```
                    +-----------+
                    |  Netlify  |  (CDN + SPA hosting)
                    +-----+-----+
                          |
                    +-----v-----+
                    |  client/   |  React 18 + Vite + TailwindCSS
                    |  (SPA)     |  Zustand + React Query
                    +-----+-----+
                          | HTTP (REST)
                    +-----v-----+
                    |  server/   |  Express + TypeScript
                    |  (API)     |  Drizzle ORM + Zod
                    +-----+-----+
                          |
                +----+----+----+
                |              |
          +-----v-----+  +----v-----+
          | Supabase   |  | Gemini   |
          | PostgreSQL |  | AI API   |
          | Auth       |  | Flash/Pro|
          | Storage    |  +----------+
          +------------+
```

## 3. Padroes de Codigo

### 3.1 Fluxo de Requisicao (Request Flow)

```
HTTP Request
    |
    v
Express Router (routes/*.routes.ts)
    |
    v
Auth Middleware (validates JWT, injects userId)
    |
    v
Zod Validator (validates request body/params/query)
    |
    v
Controller (orchestrates, transforms request/response)
    |
    v
Service (business logic, AI calls, complex operations)
    |
    v
Repository (Drizzle ORM queries, data access)
    |
    v
PostgreSQL (Supabase)
```

### 3.2 Layer Responsibilities

| Layer | Responsabilidade | Exemplo |
|---|---|---|
| **Route** | Define endpoint HTTP, method, middleware chain | `router.get('/', auth, controller.list)` |
| **Controller** | Parse request, call service, format response, handle HTTP status | `const data = await service.list(userId); res.json(data)` |
| **Service** | Business logic, validations, AI orchestration, cross-entity operations | Categorize with rules first, then AI. Calculate health score. |
| **Repository** | Pure data access via Drizzle ORM. No business logic. | `db.select().from(transactions).where(...)` |

### 3.3 Convencoes de Nomes

| Item | Convencao | Exemplo |
|---|---|---|
| Arquivos TS | camelCase ou kebab-case | `auth.middleware.ts`, `gemini.service.ts` |
| Rotas | kebab-case plural | `/api/credit-cards`, `/api/ai/health-score` |
| Tabelas SQL | snake_case plural | `credit_cards`, `categorization_rules` |
| Colunas SQL | snake_case | `user_id`, `import_hash` |
| Drizzle schema | camelCase (JS) -> snake_case (SQL) | `userId` maps to `user_id` |
| Types/Interfaces | PascalCase | `Transaction`, `CreditCard` |
| Shared types | snake_case properties | `{ user_id, bank_name }` (match SQL) |
| React components | PascalCase | `DashboardPage`, `TransactionList` |
| React hooks | camelCase com `use` prefix | `useTransactions`, `useDashboard` |
| Zustand stores | camelCase com `use` prefix | `useAuthStore`, `useFilterStore` |

### 3.4 Drizzle ORM Usage

```typescript
// Repository pattern example
import { db } from '../db/index.js'
import { transactions } from '../db/schema.js'
import { eq, and, gte, lte, desc } from 'drizzle-orm'

export async function findByUser(userId: string, from: Date, to: Date) {
  return db.select()
    .from(transactions)
    .where(and(
      eq(transactions.userId, userId),
      gte(transactions.date, from.toISOString()),
      lte(transactions.date, to.toISOString()),
    ))
    .orderBy(desc(transactions.date))
}
```

## 5. Estrategia de Error Handling

### 5.1 Backend

```typescript
// Custom error class
class AppError extends Error {
  constructor(
    public statusCode: number,
    message: string,
    public code?: string,
  ) {
    super(message)
  }
}

// Global error handler middleware (last middleware)
function errorHandler(err: Error, req: Request, res: Response, next: NextFunction) {
  if (err instanceof AppError) {
    return res.status(err.statusCode).json({
      message: err.message,
      code: err.code,
    })
  }
  // Zod validation errors
  if (err.name === 'ZodError') {
    return res.status(400).json({
      message: 'Dados invalidos',
      details: err.issues,
    })
  }
  // Unexpected errors
  console.error('Unhandled error:', err)
  res.status(500).json({ message: 'Erro interno do servidor' })
}
```

### 5.2 Frontend

- React Query handles retry logic (3 retries with exponential backoff)
- Error boundaries wrap pages for graceful degradation
- Toast notifications for user-facing errors
- Empty states for no-data scenarios

## 6. Autenticacao e Autorizacao

```
1. Client calls Supabase Auth (signUp/signIn)
2. Supabase returns JWT access token
3. Client stores token in Supabase session (auto-refresh)
4. Each API request includes: Authorization: Bearer <jwt>
5. Server auth middleware:
   a. Extracts token from header
   b. Calls supabaseAdmin.auth.getUser(token) to validate
   c. Injects userId into request
6. Role middleware checks profile.role for protected actions
```

### Roles

| Role | Permissoes |
|---|---|
| `admin` | CRUD tudo, gerenciar usuarios, configuracoes |
| `member` | Visualizar tudo, criar/editar proprias transacoes |
| `viewer` | Somente leitura em tudo |

## 7. Fluxo de Categorizacao (Rules + AI)

```
Transacao sem categoria
    |
    v
[1] Motor de Regras Deterministicas
    - Busca categorization_rules do usuario + sistema
    - Ordena por prioridade DESC
    - Testa match_type: contains, starts_with, ends_with, regex, exact
    - Se match: aplica categoria com confidence = 1.0, ai_categorized = false
    - Se nao match: continua para step 2
    |
    v
[2] AI Gemini (batch, max 50 transacoes)
    - Envia para Gemini Flash com contexto
    - Retorna: category_id, confidence, reasoning
    - Se confidence >= 0.85: aplica automaticamente
    - Se confidence < 0.85: marca como sugestao para review
    |
    v
[3] Review Manual
    - Usuario confirma ou corrige
    - Correcoes alimentam historico para melhorar IA
    - Usuario pode criar nova regra a partir da correcao
```

## 9. Rate Limiting IA

- **30 chamadas/dia por usuario** para endpoints `/api/ai/*`
- Cache de categorizacoes por hash de descricao (evita recategorizar)
- Regras deterministicas rodam primeiro (custo zero)
- Batch processing: max 50 transacoes por chamada ao Gemini
- Gemini Flash para tarefas simples, Pro apenas para chat
